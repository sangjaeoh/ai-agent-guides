# Coding Conventions

## 언제

- 타입·파일·심볼 이름을 짓거나, 새 값을 어떤 타입으로 선언할지 정할 때(배치·패키지 이름은 → [architecture](architecture.md)).
- 컴포넌트를 쓸 때. 세그먼트 배치·서버/클라 판단은 → [architecture](architecture.md)·[rendering](rendering.md), 여기선 표기·명명·접근성만.
- 스타일을 입힐 때. 새 도메인 개념의 표준어를 정할 때.
- 코드 주석·TSDoc을 쓸 때.

## 규칙

- 포맷은 Prettier, 린트는 ESLint가 강제한다. 수동 스타일 논쟁을 하지 않는다 → [code-quality](code-quality.md)의 Prettier·ESLint.

### 네이밍

- 컴포넌트 파일·export는 PascalCase, 훅은 `use`-접두 camelCase, 그 외 모듈은 camelCase.
- 파일명은 역할을 드러낸다: 컴포넌트 `Button.tsx`, 스토리 `Button.stories.tsx`, 훅 `useCart.ts`, Server Action 파일 `actions.ts`, route handler `route.ts`, 스키마 `schema.ts`, 슬라이스 public API `index.client.ts`/`index.server.ts`.
- 라우트 파일(`page`·`layout`·`loading`·`error` 등)은 `export default`로 내보낸다 — 네임드 export는 라우터가 인식하지 못한다. feature 컴포넌트는 네임드 export를 쓴다.
- 벤더 중립으로 짓는다. 백엔드·인프라 벤더명을 타입·심볼에 넣지 않는다(api-client 포트의 벤더 중립 명명은 → [data](data.md)의 api-client).
- 한 개념에는 하나의 표준어만 쓴다. 동의어를 만들지 않는다.
  - 코드·UI 카피·타입명·API 필드가 같은 개념을 다르게 부르면 드리프트가 쌓인다. 도메인 표준어는 그 도메인 슬라이스의 로컬 용어집에 둔다.

### 타입

- 백엔드 API 타입은 계약에서 코드젠한 Zod 스키마로부터 `z.infer`로 파생한다. TS 타입과 Zod를 각각 손으로 두지 않는다(단일 소스).
  - 스키마가 단일 원천이고 타입은 그 추론물이다. 이중 소스는 런타임 검증과 컴파일 타입이 조용히 어긋난다.
- `any`를 쓰지 않는다. 경계 입력은 `unknown`으로 받아 Zod로 좁힌다. 어느 경계가 무엇을 검증하는지는 → [data](data.md)의 응답 형상·api-client.
- TypeScript strict를 전제한다(설정은 → [code-quality](code-quality.md)의 TypeScript).

### 구조

- 매직보다 명시를 택한다. 과한 추상화·배럴 남용·동적 재export를 피한다.
  - 서버/클라 라인을 가로지르는 배럴 금지의 근거는 → [architecture](architecture.md)의 세그먼트와 서버/클라 구조 경계.

### 컴포넌트

- 컴포넌트의 세그먼트 배치는 → [architecture](architecture.md)의 세그먼트와 서버/클라 구조 경계.
- 재사용 디자인시스템 컴포넌트는 `packages/ui`로, feature 전용은 슬라이스 `ui`로 둔다. 도메인 로직을 `packages/ui`에 넣지 않는다. 계층 구분·승격 기준·prop API·완료 정의는 → [design-system](design-system.md)의 컴포넌트 계층·컴포넌트 API·완료 정의.
- 컴포넌트의 서버/클라 기본값과 `'use client'` leaf 판단은 → [rendering](rendering.md)의 렌더 경계.
- 접근성을 완료조건으로 만든다.
  - 시맨틱 HTML을 먼저 쓴다(`button`·`nav`·`main`·`ul`). `div` 클릭 핸들러로 버튼을 흉내내지 않는다.
  - 인터랙티브 요소는 키보드 조작·포커스 표시를 갖는다. 폼 입력은 연결된 `label`을 갖는다(placeholder를 라벨 대용으로 쓰지 않는다).
  - 정적 검사는 실제 접근성의 일부만 잡는다(강제 범위는 → [code-quality](code-quality.md)의 ESLint). 나머지는 이 판단이 소유한다.
- 이미지·폰트를 최적화한다: `next/image`(명시적 dimensions·LCP 이미지 `priority`·컨테이너 채움 `fill`)로 CLS를, `next/font`로 폰트 레이아웃 시프트를 없앤다.

### 스타일링

- Tailwind CSS v4 유틸리티 + 디자인 토큰을 쓴다. 원시 hex·px 대신 토큰을 쓴다(원시값 강제는 → [code-quality](code-quality.md)의 ESLint).
  - CSS-in-JS(styled-components·vanilla-extract)를 기각한 이유: 런타임 비용 또는 파편화된 클래스 관리. 유틸리티 + 토큰이 일관성과 제거 용이성을 준다.
- 토큰의 체계·명명·테마 분기·소유는 → [design-system](design-system.md)의 토큰.
- 조건부 클래스는 명시적 유틸(`clsx`/`cn` 등)로 합성한다.

### 코드 주석·TSDoc

- 주석은 한국어로 쓴다.
  - 도메인 개념은 그 슬라이스의 로컬 용어집에 등재된 표준어로만 부른다(용어집은 위 네이밍). 같은 개념을 주석마다 다르게 부르면 어휘가 갈린다. 미등재 도메인 개념은 용어집에 먼저 올린다.
    - 도메인 무지 계층(`shared`·`packages/ui`·`packages/config`)의 기술 어휘는 대상이 아니다.
  - 도메인 개념이 타입 doc에 처음 나올 때 `한국어(영문식별자)`로 병기한다. 멤버 doc·라인 주석에는 병기하지 않는다.
- 요약(summary)은 타입·함수·컴포넌트·훅·타입 멤버·유니온 리터럴에 둔다.
  - 타입 요약은 그 타입이 무엇인지만 적는다.
    - 어느 화면이 언제 쓰는지·왜 이 형상인지는 적지 않는다. 각각 사용처와 문서가 소유한다.
    - 개별 멤버가 하는 일은 멤버 doc이 소유한다. 같은 사실을 양쪽에 두면 각각 드리프트한다.
    - 이름이 곧 무엇인지 말하는 타입은 타입 doc이 없을 수 있다.
  - 함수 요약은, 슬라이스 public API(`index.client.ts`/`index.server.ts`로 내보내는 것)면 호출자 계약(무엇을 보장하는지)을, 내부 함수면 그 함수가 맡은 일을 적는다.
    - 정상 호출자가 마주칠 전파 예외(`redirect()`·`notFound()`·타입드 에러)는 `@throws`로 적는다. 선행조건 위반으로만 나는 예외(버그 백스톱)는 계약이 아니다.
    - Server Action·route handler 요약은 무엇을 변경하고 어떤 캐시 태그를 무효화하는지 적는다 — 호출자가 관측하는 외부 효과다(태그 체계는 → [data](data.md)의 캐시).
  - 컴포넌트 요약은 무엇을 렌더하는지 적고, 스스로 페치하거나 Suspense를 유발하면 그 사실을 더한다 — 호출자가 관측하는 계약이다.
    - 서버·클라 실행 여부는 `'use client'`·`server-only` 가드가 이미 말한다. 되풀이하지 않는다.
  - 훅 요약은 무엇을 반환하는지와 호출자가 지는 제약(클라 전용·마운트 이후 확정·정리 필요)을 적는다.
  - 타입 멤버(프로퍼티·prop·Zod 스키마 필드) 요약은 그 값이 무엇인지만 적는다. 형제 멤버와의 구분, 단위·스케일·시간대, 값 범위·존재 조건, 다른 멤버와의 산식이 대상이다.
    - 그 값이 게이트하는 동작·렌더 분기는 적지 않는다. 그 분기를 하는 컴포넌트·함수가 소유한다.
    - 멤버 요약을 Zod `.describe()`로 대신하지 않는다. 런타임 메타데이터라 사용 지점에 뜨지 않는다.
  - 유니온 리터럴·variant 요약은 의미를 적고, 상태 유니온이면 이 상태로 들어오는 조건을 더한다. 멤버당 한 줄을 넘기지 않는다.
    - 나가는 전이와 전이 그래프 전체는 백엔드가 소유한다. 프론트는 read-model이라 전이를 originate하지 않는다 → [data](data.md)의 read-model.
- 요약을 두지 않는 자리는 다른 데가 그 의미를 이미 소유하는 자리다.
  - 파일 규약이 역할을 고정하는 라우트 파일(`page`·`layout`·`loading`·`error`·`not-found`·`route`) — 파일 이름·위치. 그 라우트만의 비자명한 계약(런타임·캐시·인가 전제)만 남긴다.
  - 스키마에서 `z.infer`로 파생한 타입 별칭 — 스키마 doc.
  - public API 배럴의 재export — 원본 doc.
  - prop을 그대로 전달하는 래퍼 — 래핑 대상 doc.
    - 값을 변형·합성·기본값 주입하면 래퍼가 아니므로 요약을 둔다.
  - 테스트 케이스 — `describe`·`it` 설명 문자열. 스토리 — 스토리 이름·args.
- 이 절은 테스트·스토리 소스에도 적용한다. 테스트 케이스·스토리만 설명 문자열로 대신하고, 목·픽스처·헬퍼 타입은 똑같이 요약을 둔다.
- TSDoc은 API 계약이다. 구현 방식(How)은 적지 않는다.
  - 문서화 대상 자신의 계약만 적는다.
    - 관측 가능한 계약(전파 예외·캐시 무효화·리다이렉트)은 협력자를 거쳐 실현돼도 적는다. 협력자의 내부 동작·다른 요소의 계약은 되풀이하지 않는다.
    - 재사용 타입(공유 read-model·디자인시스템 prop) doc에는 그 값이 어느 화면에서 언제 쓰이는지 적지 않는다. 그건 그 사용처의 계약이다.
- 형식은 내용이 정한다. 요약·계약은 TSDoc 블록(`/** */`)에, 구현의 흐름·이유는 라인 주석(`//`)에 쓴다.
  - 타입 요약은 계사문(`장바구니 read-model이다`), 함수·컴포넌트·훅 요약은 3인칭 서술형(`장바구니 항목을 렌더한다`), 멤버·리터럴 요약은 명사구(`최초 등록 시각`)로 쓰고 마침표로 끝낸다.
  - 명령형(`~하라`)·장황한 주어형(`이 함수는 …을 반환한다`)은 쓰지 않는다.
  - TSDoc 블록은 에디터가 사용 지점에 계약을 띄우고 `{@link}`를 타입에 연결한다. 라인 주석은 그러지 못한다.
- 라인 주석(`//`)은 함수의 단계 표시와, 코드가 답하지 못하는 이유를 적는다.
  - 단계 표시가 먼저다. `// 1. 입력 검증`처럼 번호를 달아 흐름을 코드보다 먼저 읽히게 한다.
    - 한 단계는 한 줄 명사구로 쓴다. 단계가 둘 미만이면 달지 않는다.
    - 번호는 한 함수에 한 층만 둔다 — 상위 단계가 이미 있으면 루프·콜백 안에 다시 달지 않는다.
  - 이유는 독립된 줄에 쓴다. 왜 이 순서인지, 왜 이 값인지, 왜 손대지 않는지 등 코드가 답할 수 없는 것을 적는다.
    - 단계에 걸리는 이유면 그 단계 줄 바로 아래에 두고, 단계 줄 자체는 명사구로 유지한다.
  - 이름이 이미 말한 것을 한국어로 옮겨 적지 않는다 — 바뀌면 한쪽이 거짓이 된다. 단계 표시는 흐름을 먼저 읽히게 하는 장치라 이 금지의 예외다.
  - 시점 없는 사실로 쓴다(`우리가 …를 택했다`가 아니라 `…이므로 …한다`).
  - 주석 없이는 안 읽히는 블록은 먼저 함수·컴포넌트로 뽑고 이름으로 설명한다. 프레임워크 제약이라 뽑아도 안 읽히는 경우에만 주석으로 남긴다.
- 표기는 코드·값은 백틱, 타입·심볼은 `{@link …}`로 한다.
  - 반환·인자는 요약 문장에 녹인다. `@returns`와 자명한 `@param`은 쓰지 않는다.
  - `@param`은 비자명한 제약(조건부 존재·범위·단위)에만 붙인다.
    - 컴포넌트 prop은 prop 타입의 멤버 doc이 소유한다. `@param`으로 옮기지 않는다.
- 주석에 넣지 않는 것:
  - 결정 내러티브 금지. 무엇을 왜 채택·폐기했는지는 문서에 적는다.
  - 검사가 지키는 것 되풀이 금지. 타입·린트·포맷·경계 게이트가 잡는 걸 "지우지 마시오"류로 반복하지 않는다 → [code-quality](code-quality.md).
  - 주석 처리된 코드 금지(git 이력이 보존한다).
  - 작성자·날짜 금지(git이 추적한다).
  - TODO엔 이슈 번호를 단다(`// TODO(#123): …`).
